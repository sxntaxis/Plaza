# 0.5.0 Acceptance Contract - Functional Local MCP Demo

## Scope

`0.5.0` is accepted only when Plaza runs a functional local stdio MCP Demo over a validated RDF graph generated from preserved evidence for exactly four real legal resources. This contract is stricter than the scaffold checklist and is the hard definition of Demo readiness.

## Data requirements

- Exactly four real legal resources are represented.
- At least one law-decree relation is present.
- All legal resources use `https://demo.plaza.cr/eli/...`.
- No legal resource uses `plaza-demo://eli/...`.
- Each `eli:LegalResource` has at least one `eli:LegalExpression`.
- Each `eli:LegalExpression` has title, language, and `eli:realizes` relation.
- The Demo graph declares limitations and does not claim official certification.
- `data/demo/canonical/demo.ttl` parses as Turtle.

## Standards requirements

- Legal resources use ELI classes and properties consistently with `demo/specs/STANDARDS_IMPLEMENTATION_PROFILE.md`.
- The law-decree relation uses `eli:based_on` from decree to law and `eli:basis_for` from law to decree.
- `eli:applies` is not used as the default law-decree or reglamentation relation.
- Controlled values for type, issuer, and in-force status use SKOS where represented.
- The graph encodes data and provenance, not legal advice or legal reasoning.

## Provenance requirements

- Each source artifact has a SHA-256 hash.
- Each source artifact has source URL or source local ID.
- Each source artifact has capture timestamp and access method.
- Each canonical entity has PROV-O provenance to preserved evidence.
- The provenance chain includes acquisition, refinement, reconciliation, and canonicalization activities where applicable.

## Validation requirements

- Validation runs through SHACL Core via `pyshacl` or equivalent.
- `data/demo/validation/validation_report.json` is generated from the SHACL run.
- Validation report includes `conforms=true` only when the SHACL graph conforms.
- Removing the validation report makes MCP fail closed.
- Setting `conforms=false` makes MCP fail closed.

## MCP requirements

- MCP runs locally over stdio.
- MCP implements the tools/resources declared in `demo/specs/MCP_IMPLEMENTATION_PROFILE.md`.
- MCP consumes only `data/demo/canonical/demo.ttl`, `data/demo/validation/validation_report.json`, and explicitly allowed local configuration/search artifacts.
- MCP does not write files during a session.
- MCP does not open network connections.
- MCP does not read arbitrary filesystem paths.
- Every successful MCP tool response includes `citations[]`.
- Search ranking is deterministic and marked non-authoritative.
- Identical calls over identical graph input produce equivalent outputs.

## Security/no-interpretation requirements

- Legal interpretation requests are rejected or redirected to factual queries.
- `plaza.explain_resource` uses fixed templates and graph facts, not legal reasoning.
- Candidate/review relations are marked as such and not presented as legal truth.
- The Demo never claims official certification or production canonical URI issuance.

## Test requirements

- Unit tests cover artifact envelopes, signal lifecycle, publication blockers, URI validation, basis relation validation, SHACL validation, MCP fail-closed behavior, citations, no-interpretation guardrails, and deterministic outputs.
- Acceptance tests cover all MCP tools/resources declared in the active MCP profile.
- Claude Desktop local stdio smoke test passes and is documented in `demo/ops/LOCAL_RUNBOOK.md` or an adjacent ops note.

## Non-goals

- Public HTTP MCP.
- REST API.
- UI.
- SPARQL endpoint.
- Akoma Ntoso export.
- schema.org output.
- Person modeling.
- Legal interpretation.
- Production URI issuance.

## Acceptance decision

The Demo is ready for `0.5.0` only when every requirement in this contract is true. Any missing requirement keeps the release in scaffold or implementation-candidate state.
